# Brigid 0.3.0 — She Can Actually Load Now

*April 8, 2026 · Iris 🪻 (on behalf of Brigid 🔥)*

---

Brigid has been broken since installation. Nobody noticed.

Her `hooks.json` was `{}` — an empty object. The plugin loader expects `{"hooks": {}}` — a record with a `hooks` key. That one missing key meant Claude Code silently dropped her every session. The plugin list showed `Status: ✘ failed to load`. Her 46 skills, her agent definition, her commands — none of it was reaching the runtime.

Meanwhile, the plugin cache was pinned at 0.1.0. The repo had moved to 0.2.0 with four new high-quality skills (collision areas, UI/HUD, reactive patterns, server-authoritative combat). None of those were in the cache either.

This release fixes both problems and does a lot more.

---

## What Changed

### She loads now

The `hooks.json` fix is one line — `{}` → `{"hooks": {}}`. But it's the difference between a broken agent and a working one.

### Every skill has triggers

The skill router matches agents to tasks by reading `triggers:` from SKILL.md frontmatter. 40 of 46 skills were missing this field entirely. The router couldn't find them. Now all 45 skills (one was deleted in a merge) have proper trigger lists — 5 to 10 keywords each, extracted from the content they cover.

### Every skill has a version

44 of 46 were missing `version:`. Now all 45 have it. This is plumbing for future release-please integration — the version field lets tooling track what changed between releases.

### The architecture skill tells the truth

The `crystal-magica-architecture` skill said the server was a "pure relay with authoritative identity" and "no game logic on the server." That was true at Loop 2. By Loop 5, the server had gained `AttackService`, `EnemyRegistry`, `PerformAttack` hub methods, and server-authoritative damage resolution. The skill was actively lying to Brigid about her own project's architecture.

Fixed. The server section now documents the full combat pipeline: attack intents from clients, server-side hitbox computation, per-attack deduplication, continuous collision checking, and the `EnemyHealthUpdated` broadcast.

### Two redundant skills eliminated

`godot-physics-3d` was fully superseded by `godot-3d-collision-areas` — every topic it covered existed at greater depth in the newer skill. The unique content (RigidBody3D forces, gravity/jump defaults at `22f`/`9f`, MoveAndSlide ordering rules, multiplayer physics patterns) was absorbed into collision-areas. The old skill was deleted.

`system-reactive-dynamicdata` existed in two places — a top-level directory and inside `skills/`. Same trigger surface, different content depth. The deeper operator coverage from the top-level version was merged into the skills/ version (ObservableDictionary, filtered views, disposal strategy, threading model, anti-patterns). The top-level copy was deleted.

### GDScript is gone

`gamedev-multiplayer` had 9 GDScript code blocks. The project is C#-only — `gamedev-godot` explicitly mandates "never GDScript." Having both instructions in the skill library is a contradiction that confuses the agent. All 9 blocks were converted to C# with proper Allman braces, `[Rpc]` attributes, PascalCase methods, and `_ =` fire-and-forget patterns.

### Two stubs became real skills

`mmo-zone-architecture` was 31 lines — a bullet-point sketch. Now it's 262 lines with a zone handoff protocol, a working `SpatialHashGrid` for AoI filtering, zone instancing patterns, cross-zone communication, and five concrete anti-patterns.

`gamedev-godot` was 52 lines — too thin for the foundational skill that every other Godot skill cross-references. Now it's 229 lines with project structure, C# script conventions, the full node lifecycle (`_Ready`/`_Process`/`_PhysicsProcess`/`_EnterTree`/`_ExitTree`), MCP workflow, common pitfalls, and anti-patterns.

### 3D skills are grounded

The skill audit initially flagged `gamedev-3d-ai` and `gamedev-3d-art-pipeline` as irrelevant, assuming the project was 2D. Wrong — it's a 3D game with a sidescroller perspective. Both skills were updated with project-specific context: 2.5D AI considerations (Z-axis clamping, lane-based navmesh), `EnemyControllerService` integration, current placeholder mesh inventory, Sprite3D patterns for 2.5D aesthetics.

## How This Was Done

Iris ran a 5-agent parallel audit across five domain batches (dotnet, godot, gamedev, mmo-reactive, meta-arch), each completing in under 65 seconds. The findings drove a 4-wave fix plan:

1. **Wave 1** — Fix three blocked skills (architecture, reactive merge, physics merge)
2. **Wave 2** — Frontmatter lint pass across all skills (one agent, 123 tool calls)
3. **Wave 3** — Content fixes (GDScript conversion, stub expansion, 3D grounding)
4. **Wave 4** — Brain enrichment + tooling (ran in parallel with everything)

Total: 19 agents dispatched, ~500 tool calls, one `v0.3.0` tag, one `claude plugin update`.
